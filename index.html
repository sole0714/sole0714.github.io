1. 도입 배경 및 성과 지표

1.1 문제 인식 및 도입 목적

  • RDBMS 검색의 한계: 설계도면, 일일 보고서, 작업 지시서 등 대규모 비정형 데이터 환경에서 다중 컬럼 결합 조건 및 LIKE %keyword% 검색 수행 시 B-Tree 인덱스 효율이 급격히 저하되어 Full Table Scan 및 DB 부하 유발.
  • 검색 품질 저하: 단순 텍스트 매칭 방식으로 인해 띄어쓰기나 형태소 변형에 따른 검색 오차가 발생하고, 검색어와의 연관성 점수(Relevance Score) 측정이 불가능하여 정확도 저하.
  • 해결 방안: 내부적인 역색인(Inverted Index) 아키텍처와 Nori 형태소 분석기를 탑재한 Elasticsearch를 도입하여 고성능 전문 검색(Full-Text Search)을 구현하고 연관도 중심의 정밀한 결과 제공.

2. 클러스터 인프라 및 가용성(HA) 전략

Node Cluster 및 Scatter-Gather 메커니즘

고가용성(HA) 확보와 부하 분산을 목적으로 3대의 노드로 클러스터를 구성하여 데이터 무결성을 보장하고 자동 복구(Failover) 아키텍처를 수립했습니다.

Node ID 상태 (Status) 역할 (Role) 비고
NODE_01 ACTIVE PRIMARY 데이터 쓰기 및 읽기의 기준 노드
NODE_02 ACTIVE REPLICA 데이터 복제본 보유를 통한 고가용성 보장
NODE_03 ACTIVE REPLICA 데이터 복제본 보유를 통한 고가용성 보장
  1. 요청 수신 (Coordinating): 클러스터의 어느 노드든 검색 요청을 받을 수 있으며, 요청을 받은 노드는 클러스터 전체의 지휘자(Coordinating Node) 역할을 수행합니다. 특정 노드에 부하가 집중되지 않도록 요청을 분산 처리합니다.
  2. 검색 분산 (Scatter): 지휘자 노드는 검색 대상 인덱스의 샤드(Shard)들이 어느 노드에 있는지 확인하고, 해당 노드들에 검색 쿼리를 전달합니다. 모든 노드가 동시에 자신의 데이터를 검색하므로 처리 시간이 단축됩니다.
  3. 결합 및 정렬 (Gather): 각 노드로부터 계산된 부분 검색 결과들을 지휘자 노드가 다시 수집합니다. 지휘자 노드는 각 결과를 연관도 점수(Score)에 따라 재정렬하고 통합합니다.
  4. 결과 반환: 최종적으로 정렬된 최적의 검색 결과를 사용자에게 응답합니다.

2-1 장애 상황별 복구 가이드

  • 샤드 1개 장애 (YELLOW 상태)

    • 메커니즘: 손상된 샤드가 Primary일 경우 생존 Replica를 Primary로 즉시 자동 승격. 이후 백그라운드에서 신규 Replica 샤드를 재할당.
    • 수동 복구 API: 샤드 할당 지연 시 클러스터 리라우트 명령 수행.

3. Kubernetes (ECK) 기반 배포 전략

기존 가상머신(VM) 직접 구축 방식의 인프라 오버헤드를 해결하고자 오퍼레이터 패턴 기반의 ECK(Elastic Cloud on Kubernetes) 환경으로 아키텍처를 전환했습니다.

3-1 레거시 VM 환경 대비 개선 효과

  • 자가 치유 (Self-Healing): OOM 및 하드웨어 결함으로 파드 장애 발생 시 K8s가 이를 감지하여 자동 재시작. StatefulSet 구조를 결합하여 파드 재설정 시에도 영구 볼륨(PV)과의 1:1 매핑을 유지해 데이터 연속성 보장.
  • 선언적 수평 확장 (Scale-Out): 데이터 급증 시 인프라 설정 파일(YAML)의 count 변경만으로 신규 데이터 노드를 자동 프로비저닝 및 클러스터 편입 처리.
  • 운영 자동화: 노드 간 내부 TLS 통신 인증서 발급 및 초기 계정 설정을 자동화하고, 버전 업데이트 시 데이터 샤드를 안전한 노드로 선행 대피시킨 후 무중단 롤링 업그레이드 집행.
  • 리소스 격리: 컨테이너 단위로 CPU 및 JVM Heap 메모리 제한을 엄격히 격리하여 특정 인덱스의 과부하가 전체 물리 인프라로 전파되는 현상 방지.

4. 인덱스 설계 및 검색 최적화 전략

인덱스 매핑 구조 (construction_index)

4.2 분석기 파이프라인 세부 전략

  • Nori 형태소 분석기 (nori_tokenizer): 복합명사 분해 모드(mixed)를 채택하여 한국어 특성에 최적화된 명사 추출 및 역인덱싱 구조 확보. 조사 및 어미 변화와 관계없이 정확도 높은 토큰 추출 가능.
  • 사용자 정의 사전 (nori_user_dict): '폼타이', '시스템비계', '함바집', '헤베' 등 일반 사전에 등재되지 않은 현장 특수 전문 용어가 무의미하게 파편화되는 현상을 방지하여 검색 정확도(Precision) 향상.
  • 동의어 그래프 필터 (synonym_graph): '콘크리트-레미콘-공구리' 등 작업자 간 혼용되는 현장 은어 및 동의어를 맵핑하여 사용자가 어떤 키워드를 입력하더라도 타겟 문서를 정상 도출하는 검색 재현율(Recall) 극대화.
  • N-gram 분석기 배제 사유:
    • 글자 단위 기계적 분할로 인한 무의미한 오탐지 토큰 생성 제어.
    • 본문(content_text) 및 작업 지시문 색인 시 발생하는 토큰 팽창 및 인덱스 스토리지 낭비 억제.
    • 결론적으로 [Nori + 사용자 정의 사전 + 동의어 필터] 구조를 통해 경량 인덱스 크기를 유지하면서 고정밀 도메인 검색 엔진 완성.

5. RDBMS - Elasticsearch 동기화 아키텍처

핵심 트랜잭션 로직의 안정성 확보를 위해 애플리케이션 직접 색인 구조를 지양하고, 시스템 간 결합도를 낮춘 Logstash 파이프라인 기반 간접 색인 방식을 아키텍처로 채택했습니다.

5.1 아키텍처 검토 및 의사결정 이유

  • 애플리케이션 직접 색인: 저장 즉시 실시간 반영이 가능하나 메인 트랜잭션 내에 Elasticsearch 의존성이 결합되어 ES 장애 및 네트워크 지연 발생 시 서비스 전체 롤백 위험 존재.
  • Logstash 기반 간접 색인: 약간의 동기화 지연은 허용 가능한 도메인 요구사항을 반영하여, 메인 저장 로직과 색인 로직을 완전 분리. ES 인프라 다운 시에도 원본 데이터는 RDBMS에 안전하게 보존되며 인프라 정상화 후 재색인 가능하도록 설계하여 결함 격리 달성.

5.2 Logstash 동기화 설정 파이프라인 (logstash.conf)

input {
  jdbc {
    # feat : RDBMS 연결 및 마이그레이션 커넥터 설정
    jdbc_driver_library => "/usr/share/logstash/mysql-connector-java.jar"
    jdbc_driver_class => "com.mysql.cj.jdbc.Driver"
    jdbc_connection_string => "jdbc:mysql://localhost:3306/db_name"
    jdbc_user => "root"
    jdbc_password => "password"
    
    # feat : updated_at 컬럼 추적 기반 증분 동기화(Incremental Sync) 설정
    use_column_value => true
    tracking_column => "updated_at"
    tracking_column_type => "timestamp"
    
    # feat : 주기적 배치 스케줄링 설정 (매 분 단위 실행)
    schedule => "* * * * *"
    
    # feat : 타겟 인덱스 구조와 매핑을 일치시키기 위한 데이터 추출 쿼리
    statement => "SELECT id, project_id, source_id, document_code, file_name, content_text, site_name, work_location, is_deleted, updated_at FROM construction_documents WHERE updated_at > :sql_last_value"
  }
}

output {
  elasticsearch {
    # feat : 대상 클러스터 엔드포인트 및 인덱스 바인딩
    hosts => ["http://localhost:9200"]
    index => "construction_index"
    
    # feat : RDBMS Primary Key를 문서 ID로 매핑하여 중복 생성을 방지하고 Upsert 정합성 보장
    document_id => "%{id}"
  }
}

5.3 아키텍처 핵심 제어 포인트

  1. 완벽한 데이터 정합성 (Upsert): document_id => "%{id}" 설정을 강제하여 RDBMS 상의 식별자와 Elasticsearch 내부 문서 고유 ID를 일치시킴으로써 DB 내 수정/삭제 발생 시 중복 인스턴스가 생성되지 않고 정확히 오버라이트 처리되도록 제어.
  2. 복합 매핑 필드(Multi-field) 전략: 정렬 및 완전 일치 조건이 요구되는 file_name, site_name 등의 필드는 전문 검색용 text 타입과 정확도 필터용 keyword 타입을 동시에 가지는 다중 매핑 구조로 설계하여 검색 쿼리 유연성 확보.
  3. 가용성 보장을 위한 애플리케이션 Fallback 메커니즘: Spring Data Elasticsearch 연동 시 try-catch 구문을 통해 ES 인프라 커넥션 타임아웃 및 장애 발생 예외를 포착하고, 즉시 기존 RDBMS 검색 쿼리(RdbDocumentSearchRepository)로 라우팅을 우회시키는 차단 메커니즘을 설계하여 검색 모듈의 고가용성 구현.

🎯 결과 및 성과 (핵심 역량 및 배운 점)

  • 🏗️ 아키텍처 설계: 고가용성 클러스터 구성, ECK 기반 인프라 운영, 그리고 Logstash 파이프라인을 통한 마이크로서비스 간의 결합도 감소 및 데이터 정합성 유지 전략을 체득.
